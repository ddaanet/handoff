# Pre-activation file guards — Design

Date: 2026-05-23.

## Problem

The two handoff files have owners:

- `.claude/handoff.md` is **hook-owned** — generated by the PostToolUse
  extract hook, read into context by the SessionStart load hook. The
  agent never has a legitimate reason to touch it.
- `.claude/handoff-task.md` is **skill-owned input** — written by the
  `handoff:handoff` skill body, consumed by the extract hook.

Observed failure: the agent co-opted `handoff-task.md` as a general
scratch/todo file *before* the skill was ever triggered. Because the
existing `write-guard.sh` only checks the *path* (not whether the skill
has run), the premature write succeeded, the PostToolUse hook fired and
regenerated `handoff.md` from todo-list junk, and the artifact was
polluted with non-handoff content.

The current guard enforces *where* these files may be written, not
*when*. This design adds the *when*.

## Goal

Make both files inert outside their owner's control path:

- `handoff.md` — agent Read / Write / Edit refused **always** (fully
  hook-owned).
- `handoff-task.md` — agent Read / Write / Edit refused **until the
  `handoff:handoff` skill has activated this session**; allowed
  thereafter (the skill writes it, and the skill may read it back). The
  existing cross-project realpath guard is retained.

Reads are gated via the `Read` *tool* only. A Bash `cat`/`grep`/`sed`
bypasses a `Read` matcher; that gap is accepted. Reads of `handoff.md`
are pointless anyway (its content is injected at SessionStart), and
blocking the common path is enough — comprehensive read-blocking would
require parsing arbitrary Bash command strings, which is fragile and
out of scope.

## Activation detection: transcript scraping

The guard must answer "has `handoff:handoff` activated this session?"
without stored state.

Mechanisms considered:

- **Environment variable set by the skill/script** — *not available*.
  Each `Bash` tool call spawns a fresh shell; env does not persist
  across calls. Hooks are separate short-lived processes with no
  output channel that mutates the session environment. `settings.json`
  `env` is static config, not writable mid-session. Off the table.
- **Marker file** — wipe hooks `touch .claude/.handoff-active` at
  activation; SessionStart removes it. O(1) check, but introduces a
  stored armed/disarmed latch — precisely the state machine DESIGN.md
  congratulates itself on *not* having (PostToolUse-over-Stop). Arm and
  disarm logic would be split across `_wipe-emit.sh` and
  `load-handoff.sh` and must stay consistent. Rejected.
- **Transcript scraping (chosen)** — the guard reads `transcript_path`
  (present in every PreToolUse input) and derives the answer from the
  session JSONL each time, the same way `extract.py` derives
  files-touched. Stateless, session-scoped by construction, no new
  artifact, no lifecycle. Cost: a JSONL parse, paid only on the rare
  guarded call (the cheap basename/path check short-circuits first).

Chosen for consistency with the plugin's existing "derive from the
JSONL, store no state" stance.

### The detector — `handoff_activated()` in `_lib.sh`

Takes `transcript_path`, exits 0 if activated, 1 otherwise. Implemented
in `python3` (matching `handoff_resolve()`'s style and `extract.py`'s
JSONL parsing). Scans the JSONL for **either** activation signal —
mirroring exactly the two conditions the wipe hooks already key on:

1. **Agent path** — an assistant `tool_use` block with `name ==
   "Skill"` and `input.skill == "handoff:handoff"`. Reliable.
2. **Slash-command path** — a `type == "user"` entry recording the
   `/handoff:handoff` invocation.

Empty or missing transcript → not activated (exit 1), so a brand-new
session correctly refuses writes until the skill runs.

> **Verification gate (per CLAUDE.md testing rule).** Signal 2's exact
> shape is unconfirmed: slash commands may be stored in the JSONL not as
> literal `/handoff:handoff` text but as a `<command-name>…</command-name>`
> wrapper (observable in real transcripts). Before this detector is
> trusted, eyeball a real transcript for **both** invocation paths and
> match whatever form the slash command actually takes. Signal 1 is the
> dependable one; signal 2 must be validated, not assumed.

## Components

### `scripts/_lib.sh` (extended)

- Existing `HANDOFF_REL_TASK`, `HANDOFF_REL_OUT` constants are reused as
  the guarded paths (no new path constants needed).
- Add `handoff_activated()` as described above.

### `scripts/write-guard.sh` (extended)

Already fires on PreToolUse `Write|Edit`, parses `tool_input.file_path`
and `cwd` via `jq`, and resolves paths with `handoff_resolve()`.
Extended decision table (basename check short-circuits non-handoff
writes first):

| Target (resolved) | Activated? | Result |
|---|---|---|
| `$cwd/.claude/handoff.md` | — | **deny** — hook-owned output |
| `$cwd/.claude/handoff-task.md` | no | **deny** — inert until activation |
| `$cwd/.claude/handoff-task.md` | yes | allow |
| basename `handoff-task.md`, realpath ≠ `$cwd/...` | — | **deny** — existing cross-project guard |

Transcript path is read from the input (`jq -r '.transcript_path'`) and
passed to `handoff_activated()` only when the target is the
`handoff-task.md` path (the `handoff.md` deny needs no activation
check).

### `scripts/read-guard.sh` (new)

PreToolUse `Read` entry point. Parses `tool_input.file_path` and `cwd`,
resolves, and:

| Target (resolved) | Activated? | Result |
|---|---|---|
| `$cwd/.claude/handoff.md` | — | **deny** — hook-owned |
| `$cwd/.claude/handoff-task.md` | no | **deny** — inert until activation |
| `$cwd/.claude/handoff-task.md` | yes | allow |
| anything else | — | allow (pass-through, exit 0) |

Uses the same structured-JSON-deny / exit-0 channel as `write-guard.sh`.

### `hooks/hooks.json` (extended)

Add a `PreToolUse` matcher `"Read"` wiring `read-guard.sh` (timeout 5).

## Deny messages

Agent-facing reasons are factual state with no actionable phrasing
(agents read "do X / run Y" as instructions — see the deny-wording
convention). Examples:

- `handoff.md is generated and read by the handoff hooks; agent access is refused.`
- `handoff-task.md is inert until the handoff skill activates this session.`

Each deny also emits a curt `systemMessage` (user-facing): lowercase,
em-dash, factual — matching `write-guard.sh`'s existing style.

## Testing

Extend `tests/hook-test.sh` with synthetic PreToolUse payloads, each
carrying a `transcript_path` pointing at a fixture JSONL under
`tests/fixtures/`:

- write `handoff-task.md`, transcript with **no** activation → deny.
- write `handoff-task.md`, transcript with a `Skill` activation → allow.
- write `handoff-task.md`, transcript with a slash-command activation
  → allow.
- write `handoff.md` (any transcript) → deny.
- read `handoff.md` (any transcript) → deny.
- read `handoff-task.md`, no activation → deny; with activation → allow.
- existing cross-project `handoff-task.md` deny → unchanged.

Fixtures must mirror real JSONL shape (verify by eyeballing a recent
transcript), per the existing fixture rule. The slash-command fixture
in particular must match the real wrapper form (see verification gate).

- `DESIGN.md` — add a subsection under the guard discussion recording

- Version bump via the `release` recipe. Behavior change, but the output
  paths are unchanged, so not a breaking change.

## Non-goals

- **Blocking Bash-level reads** (`cat`/`grep`/`sed`). Out of scope;
  parsing arbitrary command strings is fragile and the injected
  SessionStart content makes `handoff.md` reads pointless regardless.
- **Re-inerting mid-session.** Activation is a one-way latch per
  session: once `handoff:handoff` has run, `handoff-task.md` stays
  writable for the rest of the session. Matches the stated concern
  ("before skill start"); a write *after* the skill ran is the skill's
  domain and re-extraction is idempotent.
